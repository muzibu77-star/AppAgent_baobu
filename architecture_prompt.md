# architecture.md 生成规范

当用户给出一个研究/工程方案，例如：

- GitHub 仓库链接或仓库名
- 方案名称
- 方案目标
- 可复用资源
- 与某篇论文、某位老师、某个研究方向的连接
- 用户希望做的改造方向

你需要基于该方案生成 `architecture.md`。输出必须使用中文，结构清晰，面向后续工程实现和论文实验设计。

## 基本原则

1. 不要编造仓库目录、文件名、数据集、模型、star 数、论文结论或实验结果。
2. 如果本地没有仓库文件，或者无法确认某个目录/模块是否存在，必须明确写：
   - “该部分需要根据仓库实际目录进一步确认”
   - “无法仅根据当前信息确认”
3. 如果用户已经提供了具体模块、目录或文件名，可以基于用户提供的信息展开分析。
4. 不要只做概念总结，必须落到：
   - 需要使用哪些模块
   - 每个模块解决什么问题
   - 为什么这个模块和当前方案相关
   - 需要如何改造
   - 训练、数据、评测分别怎么接入
5. 输出风格应接近技术方案说明，不要写成宣传文案。
6. 重点解释“现有资源如何复用”和“用户的新工作在哪里”。

# architecture.md 输出格式

## 1. 方案概述

先用 1-2 段总结用户给出的方案。

必须包含：

- 方案名称
- 原始项目/仓库
- 原项目已有能力
- 当前方案的核心改造目标
- 当前方案与用户研究目标之间的关系

示例写法：

> 本方案以 FinGPT 的金融情绪分析数据和 LoRA 微调框架为基础，引入 RETuning 的结构化推理范式，将原本简单的金融情绪三分类任务扩展为“分析框架 → 证据评分 → 反思 → 最终预测”的结构化输出任务。核心目标不是重新构建金融大模型，而是在 FinGPT 已有数据和训练 pipeline 上，验证结构化推理链与多阶段 LoRA 训练是否能提升金融文本预测能力。

## 3. 模块名称

### 3.1 解决的问题

说明该模块本身在原项目中解决什么问题。

### 3.2 数据来源 / 输入格式

说明该模块使用什么数据，输入是什么，数据如何组织。

如果无法确认具体数据集，不要编造，写：

> 当前信息无法确认该模块使用的完整数据来源，需要结合仓库中的 README、data 目录或训练脚本进一步确认。

### 3.3 训练方式 / 推理方式

说明该模块如何训练、如何推理、是否使用 LoRA、QLoRA、DeepSpeed、instruction tuning、DPO 等。

如果用户方案需要改造训练方式，要明确指出原方式和改造方式的区别。

### 3.4 评测方式

说明该模块原本如何评测，以及当前方案应该如何评测。

需要尽量落到指标，例如：

- accuracy
- macro-F1
- weighted-F1
- precision / recall
- win-rate
- preference accuracy
- ablation comparison

但不要编造已有实验结果。

### 3.5 对当前方案的意义

这一节最重要。

必须回答：

- 这个模块为什么是当前方案的起点？
- 哪些代码/数据/脚本可以复用？
- 哪些部分需要替换？
- 替换后服务于什么研究问题？
